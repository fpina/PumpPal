# Centralize Workout Builder commands and outcomes

PumpPal owns Workout Prescription creation, editing, deletion, repetition, Prescription Exercise management, Set Target management, Athlete ownership, editability, and transaction boundaries in one Workout Builder module. Mutations cross one `execute(athleteId, command)` seam whose discriminated commands use domain language; the module separately exposes the two read surfaces needed by callers: listing an Athlete's Workout Prescriptions and listing available Exercise Catalog entries.

Every state-sensitive command locks the Athlete-owned Workout Prescription before checking editability and mutating its graph. Missing or unowned subjects return `not_found`; attempts to change a historical Workout Prescription or destroy a completed Set Result return `invalid_transition`; normalized Custom Exercise name collisions and incompatible repetition tokens return `conflict`. Repeated Workout Prescriptions retain their source identity and Workout Date so an identical token retry can be distinguished from incompatible reuse. Workout Builder commands change Set Targets but never create, overwrite, or clear Set Results; those remain owned by Training Session transitions. Named successes carry only domain identities such as Workout Prescription, Prescription Exercise, or Set Target IDs. Unexpected database and invariant failures throw as operational errors so adapters can log them and return a generic server response without erasing domain rejection meaning.

SvelteKit adapters continue to own authentication redirects, route and form parsing, validation errors, redirects, and user-facing response messages. They translate persistence-shaped form fields into Builder commands and map Builder Command Outcomes through one builder-specific response helper; they do not infer ownership, editability, transaction, or repetition rules. We rejected retaining the wide route-shaped `WorkoutService` because it exposed ten mutation methods with generic exceptions, and rejected a generic form-action abstraction because validation and response data remain transport-specific.

Ordering normalization remains inside the Workout Builder implementation, while database-backed ordering invariants and set-based reorder statements are a separate persistence decision. This keeps the command seam stable while that implementation is strengthened.
